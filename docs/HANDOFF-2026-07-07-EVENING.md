## Бэклог (приоритет владельца, делать по одной, TDD)

### 1. Серийная группировка из прайса (кейс EUROHOFF/Dantex)
3-5 позиций одной серии (AVE-07M/09M/12M) должны давать ОДИН пост: одна
карточка, одни УТП, цены всех мощностей. Сейчас — 3 отдельных research+карточки.
Механика серийной подписи уже есть: `render.py::_series_header` (member_prices).
Не хватает: группировка excel_items по серии (модель без мощности/BTU —
см. `catalog/series.py` для БД-источника), один research на серию, серийное
превью. Дизайн-заготовка была в HANDOFF-2026-07-06-EVENING (Dantex-кейс).

### 2. Авто-контент «сам гонится» — показать и дать выключатель
Владелец: «постоянно гонится история с контентом под кондиционеры, не пойму,
кто запустил». **Это не сбой — это auto_tasks из config/config.yaml** (am/day/eve,
categories 2,6,7 = кондиционеры, cf-scheduler.timer каждые 5 мин, превью в
ревью-канал). Заложено с Фазы 1, владелец забыл/не знал. Сделать: показать
владельцу текущее расписание, команду `/auto` (статус/пауза/вкл) или явный
конфиг-выключатель + строку в /status. Спросить владельца, какой режим хочет.

### 3. Цены на кондиционеры стали розничными — разобраться и починить
Владелец: «должны быть оптовые, канал оптовый — цены розничные».
Подозреваемые (проверить в этом порядке):
1) свежая фича наценок источников (4d9936d, вчера: /sources, /markup,
   авто-вопрос наценки при добавлении прайса) — наценка могла примениться
   к кондиционерам из БД/прайса поверх опта;
2) `EXCEL_MARKUP_PCT` в .env на VPS (excel_run: price * (1+pct/100));
3) pricing default_markup_pct=5 + округление up_to_90 (config.yaml).
Метод: взять конкретный пост с «розничной» ценой → сверить с ценой в
прайсе/БД → вычислить фактический множитель → найти, кто его применил.
НЕ менять наценки молча — показать владельцу расчёт и спросить, как надо.

### 4. Ретрай failed с ручным фото/УТП (пошаговый wizard)
Если research не нашёл фото (боты видят «research без фото») — владелец хочет
при retry пройти ПО КАЖДОЙ failed-позиции пошагово: приложить своё фото →
свои УТП → генерация карточки в обход research. Несколько failed — очередь:
разгребаем по одной (1/N, 2/N…), потом всё уходит в работу.
Механика готова: wizard-шаги фото/УТП (`wizard_flow`), `make_card_submitter`
(submit_card с override). Расширить `/excel retry`: failed из-за «без фото» →
интерактив; остальные — как сейчас (авто-повтор).

### 5. /find и автосписок /task — пагинация результатов
Выбор «кондиционеры» показывает только первые N (лимит 30 в find_fn/_MAX_LIST,
владелец видит ~15 — вероятно, обрезка длинного сообщения Telegram), а позиций
50-100+. Нужно листание результатов (страницы или «показать ещё»), чтобы
можно было выбрать любую позицию прайса.

### 6. Сток-бот (БД oasis, склад) как источник в /task
В /task-кнопках только разделы ПРАЙСОВ. Складские кондиционеры (БД oasis,
categories 2,6,7 — исходный источник завода, `ingest/oasis_db.py`) должны
отображаться тоже: категории склада в кнопках + автосписок из БД с остатками.
Т.е. /task = единая точка: прайсы поставщиков + свой склад.
