# План: авто-выбор стиля карточки по категории (`категория → mode`)

Статус: согласовано направление, НЕ реализовано. Цель — чтобы контент-завод сам ставил
правильный стиль генерации (`mode`) на каждый товар по его категории, **детерминированно,
без ИИ и без галлюцинаций**. Следующий шаг разработки.

## Принцип (почему без галлюцинаций)
1. **Агент НИЧЕГО не классифицирует** — он рисует в том `mode`, что пришёл с задачей. Режим
   выбирается ДО генерации, на стороне контент-завода. Агенту нечего «угадывать».
2. **Режим — таблица-справочник `category_id → mode`**, не ИИ. `category_id` — реальное поле
   каталога oasis (у кондиционеров {2,6,7}; у холодильников/стиралок — свои id), проставлено
   сайтом, а не догадка. Выбор = `dict.get(category_id)`.
3. **Неизвестная категория не ломает молча**: безопасный дефолт (`mcp`) + алерт владельцу (held),
   чтобы он один раз дописал маппинг. Никаких тихих ошибок стиля.
4. **Контент карточки** (характеристики/УТП) и так детерминирован (ТТХ+УТП, пиновый {{SPECS}});
   промпт агента запрещает выдумывать/копировать с эталонов — цифры не галлюцинируют.

## Критерии (по убыванию приоритета)
- `category_id` (главный сигнал).
- Опц. уточнение по структурным полям «Группа товара» / «Наименование» (точные строки) или
  ключевым словам названия — для пограничных случаев внутри одной category_id.

## Данные (факт на 2026-07-01, кондиционеры)
`category_id`: {2: 236, 6: 4, 7: 8}. Поля-признаки: «Группа товара»=«Бытовые сплит-системы»,
«Наименование»∈{«Бытовой кондиционер»,«Облачный кондиционер»,«Тепловой насос»}, «Тип товара»=«Комплект».
(Категории бойлеров/холодильников/стиралок появятся при расширении фильтра `source.categories`.)

## Реализация (TDD)
1. Конфиг: `cards.modes_by_category: {2: mcp, 6: mcp, 7: mcp, <fridge_id>: kbt, …}` + `default_mode`.
   (Парсинг в `config.py`; ключи int.)
2. Чистая функция `mode_for(group_or_offer, modes_by_category, default) -> str`:
   `modes_by_category.get(category_id, default)`. Тест: известная категория → её mode; неизвестная
   → default; пусто → default.
3. Прокинуть в выбор режима: `scheduler.run_slot`/`cards_run` берут `mode = mode_for(g, …)` вместо
   фиксированного `slot.mode`/`default_card_mode` (slot.mode может переопределять как явный приоритет).
4. Алерт на неизвестную категорию (held + сообщение владельцу), чтобы добавил строку в карту.
5. Деплой + проверка: на кондиционерах mode остаётся прежним (mcp); смоук на синтетической «другой»
   категории → default + алерт.

## Связки
Текущее: `default_mode`/`FOTOGEN_MODE`/per-task `mode=` уже работают. Эта карта — слой автоподбора
поверх, чтобы при новых категориях не задавать `mode` вручную в каждой задаче.
