# Спека: «Контент-завод 2.0 — полный автомат» (2026-07-02)

Одобрено владельцем 2026-07-02 (чат). Контекст системы — [../../HANDOFF-2026-07-01.md](../../HANDOFF-2026-07-01.md),
[../../ARCHITECTURE.md](../../ARCHITECTURE.md).

## Зачем

Сейчас конвейер полуавтоматический: задачи ставятся вручную (`/plan` или кнопки StockBot),
таймер автогенерации карточек выключен, превью на одобрение идут в личку. Владелец хочет
**полный автомат**: система сама выбирает товары, готовит контент и карточки, шлёт на ревью
в отдельный канал; ручными остаются только (а) нажатие ✅/❌ на ревью и (б) прямые поручения
боту (Excel-прайс, «сделай 10 холодильников: 3 Beko, 3 Indesit, остальные Stinol»).

## Решения владельца (2026-07-02)

1. **УТП из интернета** — через ChatGPT фотоагента (новый тип задачи в той же очереди),
   без LLM API-ключей.
2. **Фото для Excel-товаров** — автопоиск в интернете (та же research-задача агента);
   фолбэк — генерация карточки «по названию» без исходного фото.
3. **Ревью** — отдельный закрытый Telegram-канал с кнопками ✅/❌ (вместо лички).
4. **Автомат климатики** — постоянная авто-задача в конфиге (N постов/день, окна времени),
   `/plan` остаётся для исключений.
5. **Архитектура** — вариант A: content-factory — единый «мозг» (выбор/цена/ревизия/
   публикация/state); фотоагент расширяется research-задачами.
6. **Канал публикации КБТ/холодильников** — тот же «СплитХаб.ру». Отдельная карта
   «категория → канал» не нужна: у каждой авто-задачи/задачи уже есть поле `channel`
   (пусто = канал из конфига) — этого хватит и на будущие каналы по категориям.

## Подпроект 1 — Автомат климатики (репо content-factory)

### 1.1 Авто-задачи
- `config.yaml` → секция `auto_tasks`: список `{id, filter{categories,source}, count,
  times[«HH:MM»…], mode?, confirm}` (mode по умолчанию — авто по категории, уже реализовано).
- Новый модуль `orchestrator/auto.py::materialize_auto_tasks(auto_cfgs, today, queue)`:
  на каждый конфиг создаёт Task `id=auto-<id>-<YYYY-MM-DD>`, `schedule=[«<дата> <HH:MM>»…]`
  и кладёт в очередь. Идемпотентно: `TaskQueue.add` уже `INSERT OR IGNORE` по (task_id, due_at).
- Вызов — в `scheduler_run.main()` на каждом тике (до `q.due(now)`).
- Товары кончились (все опубликованы/нет в наличии) → слот публикует 0, это норма;
  алерт не шлём (владелец видит по тишине в ревью-канале).

### 1.2 Ревью-канал
- `.env`: `TELEGRAM_REVIEW_CHANNEL_ID`; `config.yaml`: `telegram.review_channel_id` (фолбэк —
  личка владельца, как сейчас).
- `scheduler_run.build_context().confirm()` шлёт превью (карточка + подпись с ценой +
  кнопки ✅/❌) в ревью-канал.
- cf-bot (`bot/run.py`): обработка callback уже есть; добавить «уборку» после нажатия —
  `editMessageCaption` исходного превью: статус («✅ Опубликовано» / «❌ Отклонено») и снять
  кнопки. Жать кнопки может только владелец (фильтр по `from.id` остаётся).
- Публикация по ✅ — в канал из карты «категория → канал» (дефолт — «СплитХаб.ру»).

### 1.3 Автогенерация карточек
- Включить `cf-cards.timer` на VPS (код готов: `cards_run` ставит задачи с rich-specs,
  авто-ретрай ×3, лимиты per_run/max_pending). Закоммитить WIP «категория→mode».

## Подпроект 2 — Research-задачи фотоагента (репо agent_convert_foto_rituailb2b2)

- Очередь (queue.db, таблица jobs): новый вид задачи — `mode='research'`, вход без фото:
  `brand`, `model`, `category` (текст). Выход: `output/<job>.jpg` (найденное фото) +
  результат-УТП (текст, новая колонка `result_specs` или JSON рядом с фото).
- VPS API `/api/submit-job`: принять research без файла фото; `/api/next-job`/завершение —
  как у обычных задач (совместимо с failover-лизом).
- Агент (`agent.py`): обработчик research — отдельный ChatGPT-проект (`RESEARCH_PROJECT_URL`
  в `.env`, веб-поиск включён), промпт `prompts/research.txt`: «по наименованию найди
  официальное фото товара и 5–7 коротких УТП (рус., ≤40 симв.)»; Playwright скачивает
  фото из ответа, текст УТП забирает из ответа.
- Фолбэки: фото не нашлось → пометка `no_photo` (карточка потом генерится «по названию»);
  research failed → статус failed, авто-ретрай как у карточек, после исчерпания — алерт.
- content-factory читает результат так же, как карточки (read-only SQLite + output dir).

## Подпроект 3 — Excel-источник + командный язык (репо content-factory)

- **Приём файла**: cf-bot принимает document `.xlsx` от владельца → сохраняет в
  `state/prices/`, парсит (`openpyxl`; маппинг колонок в конфиге — подтвердить на первом
  реальном прайсе владельца, файл ожидается), отвечает сводкой по категориям/брендам.
- **Команда**: `/make 10 холодильники beko=3 indesit=3 stinol=*` + мягкий разбор
  («3 Beko, 3 Indesit, остальные Stinol»). Распределение: явные квоты, `*`/«остальные» —
  добор до N. Товары берутся из последнего загруженного прайса.
- **Конвейер Excel-товара** (новый source `excel`): Offer(цена из прайса × опц. наценка
  `pricing.excel_markup_pct`, дефолт 0) → research-задача (УТП+фото) → card-задача
  (mode по категории, `kbt` для холодильников) → ревизия → ревью-канал → ✅ → публикация
  с ценой. State: те же PublishState/CardJobStore + новый ResearchStore (SQLite, как CardJobStore).
- Анти-дубль: ключ = нормализованное «бренд+модель» из прайса.

## Ошибки и устойчивость

- Ревизия правилами перед каждой публикацией (без изменений).
- failed research/card → авто-ретрай ×3 → алерт владельцу, товар held.
- Идемпотентность всюду: слоты (PK task_id+due_at), публикации (PublishState),
  card/research job (store по ключу). Повторный тик ничего не дублирует.
- Оба генератора офлайн → задачи копятся в очереди; wake_agent будит вотчдог (как сейчас).

## Тестирование

- TDD, `pytest` зелёный перед каждым коммитом (сейчас 189).
- Чистые функции без сети/БД: materialize_auto_tasks, парсер `/make`, Excel-парсер,

- E2E-пилот: авто-задача на 1–2 поста в тестовый канал → владелец жмёт ✅ → пост в тестовом

## Дополнения владельца (2026-07-02, поздний вечер, после первого Excel-превью)

Первое превью Excel-конвейера (Stinol) одобрено «на 99%». Новые вводные:

   Далее: дневные лимиты генерации в конфиге, без агрессивных пересабмитов.
2. **Режим рассуждений ChatGPT medium → high** — попробовать high для research-проекта
   (владелец выставляет в UI проекта; если нет такой настройки — агенту добавить выбор
   модели, но это хрупко и ест лимиты). Трейдофф: качество УТП vs скорость и лимиты (п.1).
3. **«1%» = универсальность источников**: сайт (oasis), API поставщиков (Breez и др.),
   сообщения владельца (фото+УТП руками), другие Telegram-каналы, разношёрстные
   Excel-прайсы. Архитектурно: слой «адаптеров источников» с единым выходом
   (товар: наименование/бренд/цена/фото?/УТП?) → общий конвейер research→карточка→ревью.
   Реализация — после подпроекта 3 (Excel как первый адаптер задаёт шаблон).
4. **Единый интерфейс управления** — «пульт» контент-завода, где источники/режимы/окна/

   подпроект после 3; склоняемся к Telegram-агенту (инфраструктура ботов уже есть).
5. **Кнопка «🔄 Перегенерировать» на превью** (запрос владельца после первых вердиктов):
   третья кнопка рядом с ✅/❌. Логика: пометить awaiting, удалить файл карточки серии и

   превью придёт в ревью-группу. Для excel-товаров — пересабмит карточки с теми же
