## 1а. «Живой канал» — посты сами следят за остатками и ценами

**Зачем:** пост в «СплитХаб.ру» не должен врать: товар кончился — пометка «⛔ ПРОДАНО»,
цена изменилась — цена в подписи обновляется. Доверие B2B-аудитории + уникальность канала.

**Как:**
- **PublishState** (state, таблица `published`) расширяется колонками: `channel` (куда
  публиковали; бэкфилл существующих строк дефолтным каналом), `price` (опубликованная цена),
  `status` ('active' | 'sold'). Миграция — ALTER TABLE в конструкторе (как tries в CardJobStore).
- **Новый модуль `publish/channel_sync.py`** (чистая логика, без сети):
  `plan_sync(records, groups, pricing_cfg, min_price_delta) -> list[SyncAction]`.
  Правила:
  - серии нет в каталоге или stock=0 → action `sold` (подпись + «⛔ ПРОДАНО», status='sold');
  - status='sold', а товар снова в наличии → action `revive` (свежая подпись, status='active');
  - цена изменилась ≥ `min_price_delta` ₽ → action `reprice` (пересобрать caption через
    `render_caption` с новой ценой, обновить `price`);
  - ключи `excel|*` пропускаются (в Excel-прайсе нет остатков) — только oasis-источники.
- **Исполнитель**: `edit_caption()` в `publish/telegram.py` (editMessageCaption, ретраи как
  у publish_post) + троттлинг между правками (Telegram ≈20 правок/мин; пауза из конфига).
- **CLI `publish/channel_sync_run.py`** + systemd `cf-channel-sync.{service,timer}`
  (дефолт — 1 раз в день 08:30, до утреннего окна постинга). Конфиг `channel_sync:`
  {enabled, sold_action: mark|delete (дефолт mark), min_price_delta: 100, edit_pause_sec: 4}.
- УТП/подпись пересобираются тем же конвейером (utp_lookup Бриза) — консистентно с публикацией.

**Ошибки:** сообщение удалено руками из канала → Telegram вернёт ошибку → пометить
status='sold' (больше не трогать), алерт не слать. Сервис не падает из-за одного поста.

**Тесты:** plan_sync — чистые юниты (продан/подорожал/ожил/excel-пропуск/дельта-порог);
edit_caption — MockTransport; миграция PublishState — юнит.

## 1б. Кнопка «📩 Заказать» в постах канала

**Зачем:** превратить каждый пост в точку продажи с измеримыми лидами.

**Как (MVP без зависимости от чужих ботов):**
- При публикации в канал `publish_post` получает `reply_markup` с **url-кнопкой**
  «📩 Заказать» → `https://t.me/<order_bot>?start=ord_<key64>`. `order_bot` из конфига,
  дефолт — сам @Sendpr1ce_bot. `key64` — ключ товара, ужатый под лимит start-параметра
  (64 симв., base64 или урезанный key; маппинг key64→key хранить в state, таблица `order_links`).
- **cf-bot**: сейчас игнорирует всех, кроме владельца. Разрешить ЧУЖИМ ровно одну команду —
  `/start ord_*`: клиенту — «Спасибо! Товар: <название> — <цена>. Менеджер свяжется с вами»,
  владельцу (или в `lead_chat` из конфига) — карточка лида: @username/имя клиента, товар,
  цена, время. Всё остальное от чужих — игнор, как раньше (безопасность не ослабляем).
- Лиды писать в state (таблица `leads`) — сырьё для будущей аналитики (№9 из идей).
- Позже (не MVP): передача лида в CRM/заказ-бот с ORD-номером — когда доберёмся до их кода.

**Тесты:** сборка start-ссылки/key64 (round-trip), обработка /start ord_* (лид владельцу,
ответ клиенту, чужие прочие команды игнорируются), кнопка в publish_post (MockTransport).

## 1в. Кэш УТП и фото (база знаний research)

**Зачем:** не дёргать ChatGPT повторно (лимиты!), копить собственный контент-каталог.

**Как:** реализуется вместе с `research_pipeline.py` подпроекта 3:
- SQLite `state/research_cache.db`: `research_cache(model_key PK, brand, model, utp TEXT,
  photo_path TEXT, source 'research'|'manual', ts)`. `model_key` = нормализованное
  «бренд|модель» (lower, схлопнутые пробелы).
- Перед submit-research — смотрим кэш: есть → берём немедленно, ChatGPT не трогаем.
  Нет → research → результат в кэш (фото копируется в `research/photos/`).
- Ручные правки владельца: команда боту `/utp <модель>` + текст → source='manual'
  (manual имеет приоритет и не перезаписывается research'ем).
- Инвалидация: не нужна (ТТХ модели не меняются); удаление строки = форс-повтор research.

**Тесты:** попадание/промах кэша, manual-приоритет, нормализация model_key.
