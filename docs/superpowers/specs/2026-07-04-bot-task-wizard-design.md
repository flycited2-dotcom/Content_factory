# Spec: Кнопочный флоу постановки задачи в Telegram-боте

**Date:** 2026-07-04 · **Owner:** Алексей (владелец)

## Problem

Текстовые команды `/make`/`/find`/`/pick` требуют помнить синтаксис (число строго
первым словом, категория — одно слово без фильтров вроде «43 диагональ»). Владелец
теряет время на подбор формулировки и получает то отказ («укажите количество первым
словом»), то не тот результат (32" вместо 43", т.к. фильтр молча отбрасывается).
Кроме того, у владельца часто уже есть фото товара и/или готовые УТП (в своей базе) —
но сейчас конвейер всегда сам ищет фото и всегда гоняет research через ChatGPT, даже
когда это не нужно.

Отдельно: сегодня выяснилось, что бот использует только два слота прайсов
(`manual.xlsx`/`mail.xlsx`), а свежий прайс поставщика, приходящий владельцу в
отдельный Telegram-канал, в бота вручную не попадает — это подтверждено (поиск
«телевизор» по обоим текущим слотам даёт 1 позицию из 155+848), но это отдельная
задача (загрузка/интеграция источника), не входит в этот спек.

## Success criteria

- [ ] Владелец может поставить задачу на генерацию без единой текстовой команды:
      нажатие кнопки → выбор категории → список моделей текстом → (опц.) фото →
      (опц.) УТП → подтверждение.
- [ ] Список моделей (несколько строк, каждая — конкретное наименование) сопоставляется
      с прайсом построчно; в ответе видно, что нашлось, а что нет — без угадывания.
- [ ] Если владелец прикладывает фото — оно уходит в задачу как есть, конвейер не ищет
      фото в сети/каталоге для этой позиции.
- [ ] Если владелец прикладывает УТП текстом — конвейер не запускает research-шаг для
      этой позиции (сразу на этап карточки).
- [ ] Кнопка «Пропустить» на каждом опциональном шаге — воркфлоу не блокируется, если
      фото/УТП нет.
- [ ] Кнопка «📊 Статус» показывает то же, что сейчас `/excel` (счётчики по стадиям).
- [ ] Существующие текстовые команды (`/make`, `/find`, `/pick`, `/excel`) продолжают
      работать без изменений — кнопочный флоу это альтернативный путь, не замена.

## Scope

**IN:**
- Новая кнопка входа в флоу (постоянная reply-клавиатура или пункт меню бота).
- Пошаговый диалог с состоянием на chat_id (категория → список → фото →/пропустить
  → УТП →/пропустить → подтверждение/отмена).
- Построчное сопоставление списка моделей с обоими слотами прайсов (переиспользуем
  `search_items`/`item_key` из `excel_price.py`, по одной модели на строку, а не одна
  фраза на весь список — это НОВАЯ функция сопоставления, не расширение `/make`).
- Схема `excel_items`: два новых опциональных поля — `photo_override` (file_id/URL) и
  `utp_override` (текст); `excel_pipeline` тик учитывает их (пропускает шаги, для
  которых override уже есть).
- Кнопка «📊 Статус» как inline-алиас существующего `/excel`.

**OUT (сознательно не в этой итерации):**
- «Умный» разбор текстовых команд `/make` (число в любом месте фразы, фильтры вида
  «43 диагональ») — отдельная, более мелкая задача, можно сделать независимо позже.
- Кнопка «Добавить источник» (подключение новых прайсов/каналов) — отдельная задача,
  владелец подтвердил делать её отдельно.
- Автозабор свежего прайса из стороннего канала поставщика — тоже отдельная задача
  (см. Problem); в этом флоу фото/УТП поверх ручного списка НЕ решает проблему
  устаревших слотов прайса.
- Редактирование уже начатого диалога «назад на шаг» — в v1 только отмена и старт
  заново (`❌ Отмена` на любом шаге сбрасывает состояние).
- Работа флоу для кого-то кроме владельца (бот и так игнорирует чужие сообщения, кроме
  `/start ord_*`).

## Constraints

- Одна активная сессия флоу на владельца (он и так единственный, кто может писать боту
  вне `/start ord_*`) — состояние можно хранить одной строкой в новой таблице/файле, без
  многопользовательской изоляции.
- Telegram Bot API: фото приходит как `photo`-массив (берём последний, самый крупный) или
  `document` с image mime-type; текст УТП — обычное текстовое сообщение следующим шагом.
- Никаких новых внешних зависимостей — работаем в рамках существующего `httpx`+long-poll
  бота (`bot/run.py`), без перехода на `python-telegram-bot`/`aiogram`.
- Прод: живой бот с кнопкой «Заказать» для реальных клиентов — менять `handle_command`
  надо аккуратно, не ломая существующую маршрутизацию текстовых команд.

## Risks

| Risk | Early signal | Fallback |
|---|---|---|
| Построчное сопоставление моделей даёт много непонятных «не найдено» (опечатки, другой формат наименования у поставщика) | В тестах на реальных строках из прайса >20% строк не матчатся при точном присутствии позиции | Показывать топ-3 похожих кандидата на строку вместо жёсткого «не найдено» (as fallback UI, не в v1 логике поиска) |
| Диалоговое состояние теряется при рестарте бота (cf-bot передеплой/краш) посреди флоу | Владелец жалуется «начал вводить список — бот не помнит категорию» | Хранить состояние в SQLite (не в памяти процесса), переживает рестарт |
| Фото/УТП-override усложняют `excel_pipeline` тик (условная развилка на каждом шаге) и ломают существующий кэш `research_cache` | Существующие тесты `test_excel_pipeline.py` падают после изменения | Оверрайды — отдельная развилка ДО обычного тика (short-circuit), не трогаем существующую логику research/card для позиций без override |
