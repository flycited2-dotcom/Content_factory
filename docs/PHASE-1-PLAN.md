# Фаза 1 — план реализации (TDD)

Цель Фазы 1: **рабочий контент-завод end-to-end на пилоте (кондиционеры из oasis)** — задача от владельца
(командой боту или YAML) → выбор N товаров → цена → краткое описание → карточка (фотоагент) → ревизия
правилами → пост в Telegram-канал по расписанию. Без LLM-ключа.

Стиль: маленькие шаги, на каждый — тест(ы), держать `pytest` зелёным. Эталон — avito-bridge.

## Milestones

### M0. Каркас
- `git init`, структура пакета `src/content_factory/`, `tests/`, `pyproject`/`requirements`, `.env.example`,
  `.gitignore` (с негейтами для контент-файлов!), `config.py` + `examples/config.example.yaml`.
- Перенести из avito-bridge: `models.py`, `ingest/*`, `pricing/*`, `catalog/series.py`,
  `content/{sizing,cards,descriptions}.py`, `cards_pipeline.py`, `cards_run.py`. Прогнать их тесты.
- **Готово:** `pytest` зелёный на перенесённом ядре.

### M1. Краткое описание для Telegram
- `content/render.py::render_caption(offer|group, price, cfg)` — заголовок + 2–3 строки + цена/размеры,
  **≤ 1024 символа**, без стоп-слов, вариативность по артикулу. Поддержать переопределение из manifest
  (короткие тексты на серию).
- **Тесты:** длина ≤1024; есть бренд/цена; стоп-слова вычищены; manifest-override применяется.

### M2. Ревизия правилами (без LLM)
- `review/rules.py::review(item, cfg) -> (ok, reasons[])`: цена>0 и в границах; ≥1 ТТХ; файл карточки есть
  и не пустой; описание непустое/≤лимита/без стоп-слов; бренд/тип заполнены.
- **Тесты:** каждое правило ловит свой дефект; «всё ок» → pass; причины fail возвращаются.

### M3. Публикатор в Telegram
- `publish/telegram.py::publish_post(bot_token, channel_id, image_path|url, caption, http)` — Bot API
  `sendPhoto`. Идемпотентность: `state` хранит опубликованные ключи; повтор не шлём. Троттлинг.
- **Тесты:** через `httpx.MockTransport` — корректный вызов sendPhoto, caption ≤1024, повтор не публикуется,
  ошибки TG обрабатываются (retry/held).
- ⚠️ Реальная отправка — сначала в **тестовый канал/личку**, боевой — по ОК владельца.

### M4. Модель задач + очередь + планировщик
- `orchestrator/tasks.py` (`Task{filter, count, mode, schedule[], channel}`), `orchestrator/queue.py`
  (SQLite), `orchestrator/scheduler.py` (берёт дозревшие задачи; на каждую — выбрать N невыпущенных товаров;
  двухфазно: submit карточек / collect+review+publish дозревших).
- **Тесты:** задача порождает выбор N товаров; не публикует без готовой карточки (`require_card`);
  расписание соблюдается; не дублирует; авто-ретрай карточек.

### M5. Вход задач (A) бот-команды, (B) YAML
- `orchestrator/plans.py` — читать `tasks/*.yaml` → `Task` в очередь (см. examples/tasks.example.yaml).
- `bot/commands.py` — Telegram-бот: команда `/plan …` (и `/status`, `/cancel`, `/held`) → `Task` в очередь;
  алерты владельцу (fail-ревизии, нет фото, ошибки). Парсер команды → Task. Превью/подтверждение — опционально.
- **Тесты:** парсер команды → корректный Task; YAML → Task; невалидный вход → понятная ошибка.

### M6. Деплой + таймеры + пилот
- Деплой на VPS (tar+ssh, как avito-bridge). systemd: `cf-cards.timer` (мост к фотоагенту),
  `cf-scheduler.timer` (исполнение задач/публикация). Env `.env` (БД, фотоаген, TG-токен/канал).
- **Пилот:** задача «N кондиционеров, mode=mcp, завтра 10:00» → в тестовый канал прошёл полный цикл:
  карточка сгенерилась, ревизия ок, пост опубликован, повтор не дублируется. → перевод на боевой канал.

## Критерии приёмки Фазы 1
- Владелец ставит задачу (бот/YAML) на день/неделю; всё исполняется по расписанию само.
- Публикуются только товары с готовой карточкой и пройденной ревизией; дублей нет.
- Карточки генерятся авто (очередь+побудка+ретрай), без ручного запуска агента.
- Описания краткие (≤1024), правдивые, без LLM-ключа.
- `pytest` зелёный; секреты не в git.

## Открытые вопросы к владельцу (уточнить в начале сессии)
1. **Канал:** id/@username боевого канала + **тестового** для прогона. Токен бота — в `.env` (дать server-side).
2. **Формат поста:** что в подписи (цена показывать? размеры? контакт/призыв? хэштеги? ссылка на splithome —
   или, как в Avito, ссылку не ставим?). Дать 1 пример «идеального поста».
3. **Темп:** сколько постов/день, интервал между постами, окна времени.
4. **Первая категория после кондиционеров** и её фильтр (владелец упоминал холодильники) — какие поставщики/категории в oasis.
5. **Подтверждение перед публикацией** нужно ли по умолчанию (human-in-the-loop) или только алерт при fail-ревизии.
