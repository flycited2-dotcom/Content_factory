# Архитектура «Контент-завода»

## 1. Идея
Конвейер «товар → готовый пост в Telegram-канале», управляемый задачами. Владелец ставит задачи
(командой боту или планом-файлом), планировщик исполняет их по расписанию, каждый товар проходит
этапы: выбор → цена → описание → карточка → ревизия → публикация.

## 2. Компоненты и поток данных

```
            ┌───────────────────────── ВХОД ЗАДАЧ ──────────────────────────┐
            │  (A) Telegram-бот: команды владельца                          │
            │      «/plan 10 кондиционеры завтра 10:00,14:00 mode=mcp»      │
            │  (B) YAML-планы: tasks/*.yaml (то же, файлом)                 │
            └───────────────┬───────────────────────────────────────────────┘
                            ▼
                   ┌──────────────────┐     задачи со временем/фильтром/кол-вом/режимом
                   │   TASK QUEUE     │◄──────────────────────────────────────────────┐
                   │  (SQLite)        │                                                │
                   └────────┬─────────┘                                                │
                            ▼                                                          │
                   ┌──────────────────┐  systemd timer (каждые N минут): берёт «дозревшие» задачи
                   │   SCHEDULER /    │  и для каждой — N товаров прогоняет через конвейер ниже
                   │   ORCHESTRATOR   │
                   └────────┬─────────┘
                            ▼   на каждый ТОВАР (item):
   ┌────────────────────────────────────────────────────────────────────────────────────┐
   │ 1) SELECT   ingest из БД oasis: выбрать товар(ы) по фильтру (категория/поставщик/    │
   │             наличие), которые ещё НЕ публиковались (state) → Offer                   │
   │ 2) PRICE    pricing: опт × (1+наценка) → округление …90                              │
   │ 3) DESCRIBE content: КРАТКОЕ описание для TG (caption ≤ 1024) + ручное переопределение│
   │ 4) CARD     cards_pipeline → фотоагент: submit-job (mode) + wake; забрать готовую     │
   │             карточку из output → cards/{key}.jpg (асинхронно, с авто-ретраем)         │
   │ 5) REVIEW   правила (без LLM): цена>0, ТТХ есть, карточка-файл есть, описание ок →    │
   │             pass / fail(+причина)                                                    │
   │ 6) PUBLISH  Telegram Bot API sendPhoto(карточка, caption) в КАНАЛ; записать в state   │
   └────────────────────────────────────────────────────────────────────────────────────┘
                            ▼
                   ┌──────────────────┐
                   │   STATE (SQLite) │  что опубликовано (анти-дубль), статусы задач/товаров,
                   │                  │  card_jobs (как в avito-bridge), очередь публикаций
                   └──────────────────┘
```

### Асинхронность карточек (важно)
Карточка генерится не мгновенно: фотоагент (веб-ChatGPT на ПК владельца) обрабатывает очередь.
Поэтому конвейер **двухфазный**, как в avito-bridge:
- **Фаза submit:** на товары без карточки ставим задачи в очередь фотоагента + будим агента.
- **Фаза collect+publish:** на следующих тиках планировщика забираем готовые карточки, прогоняем ревизию,
  публикуем «дозревшие» по расписанию. Товар публикуется, **только когда карточка готова и ревизия пройдена**
  (гейт `require_card`, как в avito-bridge).

## 3. Реализация решений владельца
- **Публикация:** `publish/telegram.py` — Bot API `sendPhoto` (фото = карточка, `caption` = заголовок+
  краткое описание+цена). Бот уже админ канала. Лимит подписи **1024 символа** — описание делаем коротким.
  Идемпотентность: не публиковать товар повторно (ключ в state). Троттлинг отправки (анти-flood TG: ~20 msg/мин).
- **Ревизия — правила (без LLM):** `review/rules.py` — функции-проверки, возвращают `(ok, reasons[])`.
  Набор: цена>0 и в разумных границах; есть ≥1 ТТХ; файл карточки существует и не пустой; описание непустое,
  длина ≤ лимита, нет стоп-слов; бренд/тип заполнены. Конфигурируемо. Fail → товар в `held`, алерт владельцу.
- **Задачи:** `orchestrator/tasks.py` — модель `Task{ source/category filter, count, mode, schedule[], channel }`.
  Вход (A) `bot/commands.py` (парсит команды) и (B) `orchestrator/plans.py` (читает `tasks/*.yaml`) кладут
  в одну `TaskQueue`. Планировщик берёт дозревшие и исполняет.

## 4. Telegram-специфика
- **Bot API** (`https://api.telegram.org/bot<token>/sendPhoto`). Токен/`channel_id` — в `.env`.
- Фото: можно `multipart` (локальный файл карточки) или по URL (если карточки отдаются с splithome `/static/`).
- Подпись (`caption`, parse_mode=HTML/Markdown) ≤ 1024 символов: **заголовок** (бренд+серия+тип) +
  **2–3 строки** пользы + **цена/размеры** + контакт/призыв. Без внешних ссылок если канал это не любит.
- Расписание постинга: не чаще лимитов (≈1 пост в несколько минут на канал) — параметр в конфиге.
- На тесте слать в **тестовый канал/в личку владельца**, боевой канал — по явному ОК.

## 5. Чем отличается от avito-bridge
| | avito-bridge | Контент-завод |
|---|---|---|
| Публикация | XML-фид (Avito Автозагрузка) | Telegram Bot API `sendPhoto` |
| Единица | объявление на серию | пост на товар/серию (по задаче) |
| Описание | длинное (≤7000) | **краткое** (≤1024, caption) |
| Запуск | таймер «весь каталог» | **задачи** (что/сколько/когда) от владельца |
| Категории | кондиционеры (cat 2/6/7) | любые (старт — кондиционеры, дальше расширяем) |
| Ревизия | валидатор Avito | **свои правила** до публикации |

## 6. Конфиг (см. examples/config.example.yaml)
БД-источник, наценка/округление, фотоагент (URL/токен/очередь/output, режимы), Telegram (токен/канал/лимиты),
правила ревизии (границы цены, лимит подписи, стоп-слова), пути state, дефолтный режим карточек.

## 7. Расширяемость (потом, владелец добавит сам)
Новые категории/поставщики — через конфиг-фильтры и (при нужде) парсеры названий (как `title_parse.py`
для rusklimat). Новые стили карточек — режимы фотоагента. LLM-описания/LLM-ревизия — опциональные модули.
