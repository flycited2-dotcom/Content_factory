# Инцидент 2026-07-20: все сайты недоступны после переезда на новый VPS

Разбор и решение. Держать под рукой — симптом коварный, диагноз неочевидный.

## Симптом
Все сайты (splithome.ru, ritualb2b.ru, climat-simf.ru) не открывались **ниоткуда**:
ни с десктопа, ни с мобильного без VPN, ни с 45 проверочных точек мира
(check-host.net → везде `Connection timed out`). SSH на новый VPS тоже не шёл —
обрыв на стадии `banner exchange`.

## Что НЕ было причиной (проверено и отброшено)
- **DNS исправен.** Все три домена → `212.116.115.150`, 8.8.8.8 и 1.1.1.1 согласны,
  NS sprinthost консистентны. Первая гипотеза владельца не подтвердилась.
- **VPN на десктопе** (Happ/sing-box, TUN happ-tun, traceroute в 1 хоп) — реален и
  мешает диагностике, но не он: мир тоже не видел сервер.
- **ufw/iptables внутри системы** — владелец открыл порты, эффекта ноль.
- **Сервер сам по себе исправен**: IP на `ens3`, `default via 212.116.115.1`,
  `ping 8.8.8.8` 0% потерь, nginx слушает 80/443, 12 контейнеров Up, сайты по
  localhost отдают 301.

## Причина
**Файрвол в панели хостера Sprintbox** (`cp.sprintbox.ru` → раздел «Файрвол»).
Набор правил применён к боксам (box-891610 и box-888222); во входящих правилах был
разрешён **только TCP 8080**. Всё остальное резалось ДО операционной системы.

## Решение
В панели → «Файрвол» → «Входящие правила» → добавить TCP **22**, **80**, **443**
(Все IPv4 + IPv6). Порт **8080 не удалять** — на нём `crm-nginx`.
Результат: сайты сразу отдали 200, SSH заработал.

## Диагностическая памятка
- Асимметрия «наружу можно, внутрь нельзя» при живых nginx/маршрутах = **внешний**
  файрвол, а не система.
- `banner exchange timeout` = блок ДО аутентификации; ни ключ, ни пароль не помогут.
- Проверять доступность только независимым инструментом (check-host.net) и **всегда
  делать контрольный тест** самого инструмента: внешний прокси-API отдавал 522 даже
  на example.com, то есть врал.
- Внутрь сервера при мёртвом SSH — только VNC-консоль хостера.
